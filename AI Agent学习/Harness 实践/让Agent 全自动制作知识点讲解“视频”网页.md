> 本视频详细介绍了如何通过 Harness 工程实践，利用 Agent 自动化将技术文档转化为精美的知识讲解视频。作者并未依赖现有的视频生成模型（因其不可控），而是选择通过 Web 技术构建可控的自动化流程。该方案将内容创作划分为四个核心阶段，引入了严格的开发规范、多阶段的人工检查点以及并行开发机制，确保在不同主题和文章下都能稳定产出高质量、节奏自然且视觉统一的视频内容。

## **视频制作的要点**
1、把书面文章变成口播稿（感觉在跟人聊天）
![[Pasted image 20260710032659.png]]2、把口播稿拆成开发计划/开发大纲
![[Pasted image 20260710032826.png]]3、把文字的抽象概念用动态的效果演示出来
![[Pasted image 20260710032940.png]]4、演示与口播节奏对其
![[Pasted image 20260710033011.png]]
我们可以把以上的任务交给Agent来做，但是直接把材料丢给Codex做，他可能直接跑偏。模型的能力很强大，但是他需要一套机制来驾驭这个过程——Harness驾驭工程

接下来我们要用`web-video-presentation`这个skill作为Harness的实践案例

Skill用来做什么？（点击打开）
skill本身就是给Agent看的操作手册，他告诉agent什么时候能做什么时候不能做。当Agent加载skill后就不用每次重新设立规矩再干活。

一个成熟的skill需要包含，六个核心：
1. 上下文管理：模型已知的信息，看到了什么
2. 工具系统：模型到底能使用什么工具
3. 执行编排：下一步模型需要做什么，设置计划表
4. 状态与记忆： 系统如何跨步骤的同时保持连续性
5. 评估与观测： 约束与恢复： 出错了怎么办，怎样避免跑偏

### 执行编排
> 让模型知道下一步该做什么

整个流程有四个阶段和两个人工检查节点

阶段一：内容编写
AI要在这个阶段同时产生出两个东西，口播稿`script.md` 和`outline.md` 
![[Pasted image 20260710050139.png]]
口播稿决定信息的描述方式和整体节奏，开发计划决定每章做几步，每步放什么内容。

**Checkpoint Plan节点**
- [ ] 稿子和开发计划检查修改
- [ ] 主题选择
- [ ] 素材准备
- [ ] 章节使用并行还是串行开发

### 开发阶段
开发阶段并不是一次性完成的，而是经过第一章节开发之后需要经过人工来验收，后续开发需要以这个章节为基准
![[Pasted image 20260710170109.png]]
每个章节在完成之后都要对照检查清单进行自检。

**Checkpoint Audio节点**
检查是否需要合成音频，如果不需要则进行下一阶段

## 上下文管理
主要解决的是模型能看到什么的问题。
文中`web-video-presentation`skill 工作流横跨了四个阶段，每个阶段涉及了很多复杂的流程，如果在启动的时候一股脑地把文档全部输入，模型的注意力就会被严重的稀释。所以我们要把所有的信息拆分成很多份文档，每一份文档只在指定阶段读取，这样做模型面前摆的都是当前最重要的信息。
![[Pasted image 20260710172025.png]]

## 状态和记忆
系统怎么样跨多个步骤还能保持连续性和稳定性？
主要靠这个`web-video-presentation`skill里的开发计划`outline.md`。一个项目可能有是几个章节，上百个步骤，Agent写到后面章节的时候，已经把前面定的结构和用户的修改忘得一干二净了。开发计划的作用就是把这些关键的信息固定下来，成为开发阶段的持续记忆。

![[Pasted image 20260710182159.png]]
在开发阶段Agent除了会读取`outline.md`开发计划，还会强制读取两个文件`script.md`口播稿（负责整体节奏）`article.md`原文章（负责信息的密度，画面的案例和数据）。
![[Pasted image 20260710183214.png]]
本质上这是一种文件化的工作记忆，把关键的状态步骤写进文件，需要的时候再让agent读取。
## 工具系统
> 工具系统解决的是模型能做什么的问题，本skill主要用的是agent自带的文件读写的工具。

在章节具体的开发流程中存在一个关键的问题，就是多个Agent同时修改一个项目的时候会不会相互产生冲突？
skill为此设置了严格的隔离机制，每一章设置独立文件夹，独立的CSS，主题走全局变量保证风格一致。
![[Pasted image 20260710195226.png]]

## 约束与恢复
所有Agent系统都绕不开自评失真的问题，为解决这个问题需要使用自检-修复-再汇报的规则
![[Pasted image 20260710200014.png]]
每个阶段都有一套完整的自检规则，以下介绍三种自检的执行方式
![[Pasted image 20260710200653.png]]
**Agent Team（最优方案）**
使用agnet teams开个独立的reviewer agent 产出+清单，从零核查
![[Pasted image 20260711214510.png]]


**SubAgent（其次）**
使用子代理，没有Teams能力时退而求其次，子代理走同样的清单流程
![[Pasted image 20260711214415.png]]

**当前Agent自检（兜底）**
自己核查自己--但必须严格逐项，不允许目测一遍就放行

![[Pasted image 20260710201954.png]]
